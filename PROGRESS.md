Development progress log for cluster-whisperer. Tracks implementation milestones across PRD work.

## [Unreleased]

### Fixed
- (2026-05-11) Fixed a spurious "Failed to obtain server version" warning from the Qdrant client that appeared on every sync run. The cause: the Qdrant JS client was initialized with `host` and `port` separately, which caused it to construct a URL with an explicit `:80` suffix. NGINX ingress routes on hostname only and failed to match requests with `Host: qdrant.nip.io:80`, so the version check fell through to the error handler. Switching to the `url` parameter passes the URL as-is, the Host header matches the ingress rule, and the check now succeeds.
- (2026-05-11) Fixed the Chroma collection verification in `setup.sh` incorrectly warning "Could not verify Chroma collection — check manually" even when data was present. Two issues: (1) the API path was wrong (`/api/v2/collections` returns 404 in Chroma 1.x; the correct path includes tenant and database: `/api/v2/tenants/default_tenant/databases/default_database/collections`); (2) the NGINX ingress may not be ready the instant after `kubectl apply`, so the first request could race. Added a 3-attempt retry with 5-second delays.
- (2026-05-11) Added 7 ephemeral Kubernetes access-review resources (`bindings`, `localsubjectaccessreviews`, `selfsubjectaccessreviews`, `selfsubjectreviews`, `selfsubjectrulesreviews`, `subjectaccessreviews`, `tokenreviews`) to the explicit exclusion list in the discovery pipeline. These create-only resources have no list verb and were already being filtered implicitly; making the exclusion explicit documents the intent.

The deck has three sections: Section A explains the agentic loop with a progressive diagram (Developer → Agent[LLM] → Tools → answer); Section B shows the RAG pipeline that lets the agent search 360+ Kubernetes CRDs by semantic meaning instead of brute-force enumeration; Section C is a two-act guardrails sequence — the happy path (ManagedService gets approved) and the sad path (Tron gets blocked by Kyverno) — each told as a 8-slide dramatic unroll with the characters "talking" via italic text in the boxes. - (2026-05-07) `setup.sh` now recovers cleanly when an existing GKE cluster is found but the local kubeconfig is missing or stale — it regenerates credentials via `gcloud container clusters get-credentials` before proceeding, rather than immediately erroring out with a confusing "run teardown.sh first" message.
- (2026-05-07) All Kyverno admission webhooks are now set to `failurePolicy: Ignore` during setup, not just the primary resource webhook. Kyverno installs with `forceFailurePolicyIgnore=true` (which only covers the resource webhook), then five additional webhook configurations are explicitly patched: policy-validating, CEL-exception, exception, global-context, and cleanup. This prevents a deadlock where Kyverno's webhooks with `failurePolicy: Fail` blocked GKE's Konnectivity agent from re-authenticating its control-plane-to-node tunnel during Kyverno restarts, causing "No agent available" errors on all subsequent kubectl operations.
- (2026-05-07) `setup.sh` now probes the Kubernetes API server directly (`kubectl version --request-timeout=5s`) before retrying failed Helm installs, in addition to checking gcloud cluster status and waiting for GKE operations. This handles a case specific to GKE + Crossplane: registering 300+ CRDs OOM-kills the managed API server, which GKE restarts silently — `gcloud container clusters describe` reports RUNNING and no RESIZE_CLUSTER operation appears, but the API server is actually unreachable. The kubectl probe is the only reliable ground truth.
- (2026-05-07) Kyverno is now installed with three additional Helm flags tuned for Crossplane environments: API client rate limits raised to 100 QPS / 100 burst (default 20/50 is exhausted by Kyverno's startup API discovery walk over 300+ CRDs, causing the throttling-restart loop); a `matchConditions` CEL expression preventing the API server from calling the Kyverno webhook for CRD registrations at all (reducing load during the Crossplane provider flood); and a `resourceFiltersInclude` entry skipping internal Kyverno evaluation of CRD objects. Community research confirmed the rate limit and matchConditions approaches as the documented fixes for this Kyverno + large-CRD-count failure mode.
- (2026-05-07) Kyverno admission controller is now restarted (`rollout restart`) after Crossplane registers its 300+ CRDs, before the `ClusterProviderConfig` is applied. Kyverno v1.17.1 only discovers API resources at startup — any CRD registered after Kyverno starts is unknown to it, causing admission requests for those types to fail with "resource not found in group." A restart forces re-discovery. Without this, `ClusterProviderConfig` application failed on every cluster run.
- (2026-05-07) Health check retry loops now show the actual error on every attempt, not just at the end. curl exit codes are decoded to human-readable descriptions ("Failed to connect to host" instead of "exit 7"). `run_helm_step` prints the last 5 lines of the failed command's output inline rather than logging a temp file path. The capability inference and instance sync CLI pipelines no longer print one line per resource (843+ lines) — they show only warnings and a final completion count.

- (2026-05-06) GKE cluster setup now reliably installs Kyverno even when the control plane is busy.
